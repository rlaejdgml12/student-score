# 2026-07-22 학습 노트

## 객체지향 (OOP)
객체 = 상태(필드) + 행위(메서드)를 하나로 묶은 것. 절차지향은 데이터와 계산이 따로 놀아서, 요구사항이 바뀌면 계산 로직이 흩어진 여러 곳을 다 찾아 고쳐야 함. 객체지향은 계산을 객체 안 메서드 한 곳에 모아두므로, 바깥에서는 `student.getTotal()`처럼 메시지(메서드 호출)만 보내면 됨 → 고칠 곳이 한 군데로 모여 유지보수가 쉬워짐.

## 생성자
- 기본 생성자: 직접 만들지 않으면 컴파일러가 매개변수 없는 생성자를 자동 생성 (필드는 String→null, int→0으로 초기화됨)
- 단, 매개변수 있는 생성자를 하나라도 직접 만들면 자동 기본 생성자는 더 이상 생성되지 않음 → 필요하면 직접 만들어야 함
- 오버로딩: 같은 이름의 생성자를 매개변수 개수·타입만 다르게 여러 개 두는 것
- `this()`: 생성자 안에서 다른 생성자를 호출 (반드시 첫 줄). 초기화 로직 중복을 줄여줌
- `this.필드 = 매개변수`: 왼쪽 `this.필드`는 생성되는 객체 자신의 필드, 오른쪽은 생성자가 받은 매개변수

## 패키지
클래스를 폴더처럼 묶어서 정리하는 것. 패키지를 나누는 목적은 유지보수·정리정돈이고, `import`는 다른 패키지에 있는 클래스를 쓸 때 "정확히 어느 패키지의 어떤 클래스인지"를 컴파일러에게 알려주는 연결 통로. 같은 패키지 안에서는 import 없이 서로 접근 가능.

## 접근 제어자 (실습 중 직접 겪은 문제)
필드에 접근 제어자를 안 쓰면 default(같은 패키지에서만 접근 가능) 상태가 됨. `domain`/`app`으로 패키지를 나누자 `student1.name`이 다른 패키지에서 접근 불가 에러 발생 → `getName()` 메서드로 우회. 이는 내일 배울 캡슐화(private + getter)로 이어지는 문제.

## 오늘 실습 — StudentScore v2
- `Student`를 `domain` 패키지로, `StudentMain`을 `app` 패키지로 분리
- 생성자로 이름·국어·수학·영어 점수 초기화 (`this()`로 기본 생성자가 매개변수 생성자 재사용)
- `getTotal()`, `getAverage()` 메서드로 계산 로직을 `Student` 안으로 이동 (`getAverage()`는 `getTotal()`을 재사용)
- `main`에서는 `+`, `/` 연산 없이 메서드 호출로만 처리
